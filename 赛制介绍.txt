Ubiquant Challenge-AI推理挑战赛 | 赛制介绍
一、背景介绍
在推理系统的真实生产环境中，模型能力只是基础，系统如何在有限资源下处理动态请求，才是决定性能与成本的关键。
本次挑战赛构建了一个高度抽象但贴近现实的推理服务场景：  平台持续产生带有不同时延要求（SLA）、精度需求与收益权重的推理任务，参赛者需要设计并实现完整的推理服务系统，在动态流量中进行实时决策与调度。
参赛者的目标是在有限 GPU 资源约束下，最大化整体收益。
二、任务场景
平台会持续生成推理任务流（Task Stream），每个任务包含：输入请求（Prompt / Context）、SLA（时延约束）、精度要求（如输出质量或正确率）、收益（Reward）
任务将以流式形式广播给参赛系统。参赛者需要完成以下决策：是否接收任务、任务执行时机、批处理策略、资源分配、是否进行缓存复用
任务若未在 SLA 内完成，将被视为失败并产生惩罚。
2.1 单题收益
在 SLA 内完成的任务，收益由以下因素综合决定：
- 任务难度：不同任务所需的推理计算量不同，计算量越大的任务收益越高。generate_until 类任务（需要完整生成文本）的收益显著高于 loglikelihood 类任务（仅需计算概率）。
- SLA 等级：SLA 要求越严格的任务收益越高。高 SLA 等级意味着选手需要更强的系统优化能力才能在时限内完成。
- 采样参数：对于 generate_until 类任务，更复杂的采样参数（如高惩罚系数）会增加推理难度，对应更高的收益。
- 正确性：选手的推理结果将与参考模型的结果进行比较，正确性越高收益越高。正确性为零的任务不获得任何收益。
2.2 正确性评判
正确性得分的评判方式因请求类型而异：
- generate_until 任务：选手的生成结果与参考模型的结果进行比较。
- loglikelihood 任务：平台根据选手返回的各候选答案 logprob 选择 argmax，与参考模型的 argmax 进行比较。答案一致得满分，不一致得零分。
- loglikelihood_rolling 任务：以选手计算的 word_perplexity 是否在参考值的合理范围内判定。
参考模型答错的样本不会出现在评测中。
2.3 任务完成状态
每个任务从分配给选手起，存在一个绝对超时时限。在该时限内，任务可能处于以下状态：
状态
说明
得分影响
SLA   内完成
在 SLA 时限内提交了有效结果
获得正常收益
SLA   超时完成
超过 SLA 但在绝对超时内提交了结果
不得分，不扣分
未完成
绝对超时后仍未提交结果
扣分
系统崩溃
连接断开、返回错误码、进程异常退出
扣分
绝对超时按任务类型有所不同——长文本类任务的超时时限更长。
2.4 反作弊机制
平台会对异常提交行为进行检测。提交无意义回复（如空字符串、固定内容、连续错误）的行为将被视为恶意提交，触发额外惩罚和信誉降级。
2.5 总分
Score = 成功任务收益之和 - 未完成惩罚之和 - 恶意提交惩罚之和 - 崩溃惩罚之和

三、赛制设计
3.1 赛程安排
比赛分为三个阶段：
（1）热身阶段（Warmup）
- 提供 baseline 推理服务框架及示例代码
- 提供任务流回放工具
- 不计入排名
（2）预赛阶段（Offline Evaluation）
- 参赛者离线开发并优化推理系统
- 平台提供标准任务流（部分公开，部分隐藏）
- 参赛者提交代码
评测方式：
- 使用统一测试环境运行参赛系统
- 输入固定任务流
- 记录系统行为与收益
- 按 Score 排序，前 N 名晋级决赛
预赛计分环节的题库与练习环节不同，请勿依赖特定题目进行针对性优化。
（3）决赛阶段（Online 对抗）
决赛采用实时对抗机制，所有入围队伍同时参与，在统一资源约束下进行在线推理服务竞争。
任务分发机制：
- 平台维护全局任务队列，任务持续动态产生
- 所有队伍可实时查询任务队列，并自主决定是否参与“抢单”
任务分配规则（Bid-based）：
平台基于队伍提交的信息进行综合评估，按照以下原则进行任务分配：
- 每个任务最多分配给 1 支队伍
- 平台根据 Bid 综合指标排序，选择最优的队伍获得执行资格
执行与计分规则：
- 每个任务分配给 1 支队伍执行
- 平台通过持续出题保证任务供给充足，每支队伍都有充分的任务可接
- 在 SLA 内完成任务 → 获得对应 reward
- 计分规则与预赛一致（第三节评分规则）
Bid 信誉机制：
- 获得任务分配后未在规定时间内提交结果的，视为 Bid 虚报
- SLA 超时但最终提交了正确结果的，不算虚报
- Bid 虚报将立即降低队伍信誉分，影响后续一段时间内的任务分配优先级
- 信誉分可随时间自动恢复
实时排名机制：
- 平台根据各队伍累计收益进行实时排名更新
- 排名信息将采用延迟展示机制（如延迟若干秒），以防止策略抄袭与博弈干扰
四、系统要求
参赛者需实现完整推理服务系统，包括但不限于：
4.1 推理调度
- 多任务调度策略
- SLA 感知调度
- 优先级管理
4.2 动态批处理
- 自适应 batch size
- 长短请求混合处理
- 吞吐与延迟平衡
4.3 缓存与复用
- KV Cache 管理
- Prefix Sharing
- 重复请求优化
4.4 资源管理
- GPU 利用率优化
- 显存管理
- OOM 避免与恢复
五、容错与稳定性要求
为模拟真实生产环境，系统需具备基本容错能力：
- 网络中断后可恢复运行
- 请求具备唯一 ID，需支持幂等处理
- 重复任务不得重复计分
- 系统异常需自动恢复（推荐）
六、资源与限制
为保证公平性，所有参赛队伍使用统一资源配置：
- 固定 GPU 数量与型号
- 固定 CPU / 内存限制
- 固定模型权重（由平台提供）
禁止：
- 使用额外外部计算资源
- 修改模型结构或权重
- 调用外部推理服务
七、评测重点
本次比赛重点考察：
- 推理系统整体设计能力
- 在线调度与决策能力
- 系统性能优化能力
- 资源管理与工程实现能力
八、总结
本次挑战赛旨在还原真实推理服务环境中的核心问题：  在不确定、动态变化的请求流中，如何通过系统设计实现性能、成本与收益的最优平衡。
参赛者需要的不仅是模型能力，更是对系统的理解与工程实现能力。
